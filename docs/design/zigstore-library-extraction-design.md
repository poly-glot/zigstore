# Design: extract `zigstore` — a reusable embeddable + networked record store

## Decisions at a glance

| Decision | Choice |
|---|---|
| **What to extract** | BROAD cut: storage + WAL + snapshot + epoll server + binary-protocol framing + TS-client codegen — everything generic, ~11k loc across every seam. |
| **Sequencing** | Big-bang to a single `v1.0.0`, landed as **one validated PR** (no extended broken `main`). |
| **Engine API** | Hybrid: comptime-generic data/codec/codegen plane (`Engine(schema)`) + runtime interfaces at the dynamic seams (dispatch callback, recover hooks, periodic workers, snapshot host). |
| **Audience** | Public / OSS-quality. Name: **`zigstore`** (`github.com/poly-glot/zigstore`). |
| **Comment policy** | Doc comments (`///`, `//!`) on the **public API surface only**; internal code stays comment-free. Overrides the global no-comment rule for this repo only. |
| **First consumer** | `dmozdb` (this repo). It validates the engine end-to-end before `v1.0.0` is tagged. |

## Context & goal

`dmozdb` is a Zig backend (custom binary protocol, B+Tree paged storage, WAL, epoll
server) powering a DMOZ-style web directory. The backend is a flat module graph rooted
at `src/main.zig` with relative `@import`s — no module boundaries.

The goal is to lift the generic database/server machinery into a standalone, semver-tagged
public library, **`zigstore`**, consumed by `dmozdb` and future apps with similar
requirements (binary-protocol record-store services with SSR/TS clients) as a
`build.zig.zon` dependency. A dependency-acyclic boundary is mandatory: **no engine file
may import or reference an app file or type.**

The B+Tree already compares opaque `[]const u8` keys, so the reusable core is genuinely
*"a fast embedded ordered byte-key/byte-value store with WAL durability, crash recovery,
page cache, a B+Tree, an epoll reactor, a binary-protocol framing/TLV codec, and
reflection-driven client codegen."* The hierarchy that defines the *product* (slugs,
subtree counts, breadcrumbs) is entirely application semantics layered on top.

## §1 · Engine identity & scope

`zigstore` is an embeddable **and** networked ordered-record store. It knows nothing about
categories, links, slugs, or hierarchy. The BROAD scope is justified — not speculative —
because a concrete second consumer (another binary-protocol service with an SSR client) is
intended, giving the MIDDLE/BROAD abstractions a real migration target rather than a single
hypothetical one.

The engine surface:

- **Storage**: 16 KB slotted pages, sharded LRU page cache, on-page freelist, a generic
  ordered byte-key/byte-value B+Tree (point, range, insert, delete, crash repair).
- **Durability**: WAL writer + replay, memtable + bloom filter, snapshot/checkpoint,
  monotonic-sequence commit ordering.
- **Server**: epoll reactor, connection buffers, signal handling, a binary-protocol
  framing/TLV codec, latency histograms, a generic bootstrap/run loop.
- **Codegen**: reflection-driven TypeScript-client emitter, parameterized over the app's
  field-type mapping.

## §2 · Boundary — file manifest

### Moves as-is (zero back-edges in the dependency graph today)

`page.zig`, `page_cache.zig`, `freelist.zig`, `btree/btree.zig`, `btree/btree_insert.zig`,
`btree/btree_delete.zig`, `btree/btree_search.zig`, `btree/btree_repair.zig`,
`btree/btree_helpers.zig`, `wal/wal.zig`, `wal/wal_replay.zig`, `memtable.zig`, `bloom.zig`,
`inverted_index.zig` (tokenizer), `histogram.zig`, `connection.zig`, `signal.zig`.

These import only each other (or nothing) from `src/` and carry no DMOZ tokens in code —
only in test fixtures. They copy out and have imports repointed.

### Split — the generic half lifts, the DMOZ half stays

| Source | Engine half | App residual |
|---|---|---|
| `types.zig` | `codec.zig`: `Serializable`, `FixedString`, `CompositeKey`, `encodeU64`/`decodeU64`, neutral `hash` | `schema.zig`: `Category`, `Link`, `LinkStatus`, `RepairOp`/`RepairTask`, `ParentChildKey`/`CategoryLinkKey`/`SubmitterLinkKey` + size-assert tests |
| `changeset.zig` | `wire_codec.zig`: `encodeStruct`/`decodeStruct`/`encodeField`/`decodeField`, version framing | residual `Op`/`Effect` structs/`ChangeSet` union |
| `file_header.zig` | generic superblock (magic/version/page_size/free_list_head/page_count/seq + comptime-generated per-index `<name>_root`/`<name>_count`) | none — the DMOZ catalog *is* generated from the app's schema |
| `database.zig` | `Store` (generated by `Engine(schema)`): file/header/cache/free_list/wal/snapshot + named trees + named memtables + flush/drain/recover scaffolding + `op_latency` + apply ordering | `Directory`: composes `Store`; owns `next_*_id` allocators, `url_bloom`, `subtree_cache`, status counts |
| `snapshot.zig` | generic over a `SnapshotHost` interface | none |
| `commit.zig` | generic over a record type (WAL append + seq-gated apply + durability wait) | the app's `serialize`/`apply` callbacks |
| `binary_protocol.zig` | `protocol/framing.zig`: header consts, `writeResp`/`writeErrorResp(Sub)`, the comptime TLV codec, `readOptionalString`, `writeRowList`, the pipelined frame loop, base `Status` | residual `Op` enum, DMOZ `Status`/`SubStatus` extensions, `mapErrorWithSubStatus`, `dispatch`, all `handle*` fns |
| `main.zig` | `server_config.zig` (`ServerConfig`: port/bind/cache/threads/wal_batch/snapshot interval/trust-list) + `run.zig` (generic bootstrap) | residual `main`: DMOZ config fields, `DMOZDB_*` env keys, `Top`/`Lost and Found` branding, concrete `Directory` instantiation |
| `gen_client_ts.zig` | `tsgen.zig`: the reflection emitters, parameterized over a field→TS-type/reader table | residual `main()` driver, chosen DMOZ records, output path, `writeLinkStatusEnum` |
| `epoll.zig` | generic reactor over an opaque ctx + a `Handler` (`processFrames` + `HEADER_SIZE`) + `ServerConfig` | none |

### Stays in `dmozdb` (the first consumer)

`operations/*`, `apply/*`, `verifier.zig`, `subtree.zig`, `repair/*`, `wal/wal_apply.zig`,
`import_main.zig`, plus the residual halves above and the split `e2e_test.zig`
(DMOZ-workflow tests stay; infra roundtrip tests move to the engine).

## §3 · The hybrid API

### Comptime data plane — `Engine(schema)`

The app declares its store as a comptime schema; the engine generates the typed storage
layer. This promotes `database.zig`'s existing `tree_fields` comptime table (index name →
header root field → header count field, walked with `inline for`) into the public API.

```zig
const zigstore = @import("zigstore");

const schema = zigstore.schema(.{
    .magic = 0x444D4F5A,          // app-chosen; the engine no longer hardcodes 'DMOZ'
    .format_version = 6,
    .indexes = .{
        .{ .name = "categories_by_id",        .key = .u64 },
        .{ .name = "cat_by_parent",           .key = .{ .composite = &.{ "parent_id", "child_id" } } },
        .{ .name = "links_by_id",             .key = .u64 },
        .{ .name = "link_by_category",        .key = .{ .composite = &.{ "category_id", "link_id" } } },
        .{ .name = "link_by_url_hash",        .key = .u64 },
        .{ .name = "link_by_submitter",       .key = .{ .composite = &.{ "submitter_id", "link_id" } } },
        .{ .name = "categories_by_slug_path", .key = .bytes },
        .{ .name = "categories_by_slug_only", .key = .bytes },
        .{ .name = "categories_index_tree",   .key = .bytes },
        .{ .name = "links_index_tree",        .key = .bytes },
        .{ .name = "slug_path_repair_queue",  .key = .u64 },
    },
    .memtable_indexes = &.{
        "categories_by_id", "cat_by_parent", "links_by_id",
        "link_by_category", "link_by_url_hash", "link_by_submitter",
    },
    .counters = &.{ "next_category_id", "next_link_id", "next_repair_seq" },
});

pub const Store = zigstore.Engine(schema);
```

From `schema`, the engine generates at comptime:

- `Store.Header` — the superblock with a `<name>_root` and `<name>_count` slot per index, a
  persisted slot per declared `.counter`, plus the generic magic/version/page_size/
  free_list_head/page_count/seq fields. **No DMOZ field names; the app supplies `magic`.**
  (This resolves the `file_header` blocker via the type-safe-roots option.)
- The `Store` struct — cache/free_list/wal/snapshot, the named B+Trees, the named
  memtables, the apply-ordering machinery, and the `op_latency: *[256]AtomicHistogram` sink.
- Generic `init` / `flushHeader` / `drainOneInner` — today's `inline for (tree_fields)`
  becomes engine-internal, driven by the schema.

### Runtime seams — interfaces and callbacks

The dynamic seams stay runtime; comptime there would over-couple.

```zig
// recover: engine owns WAL replay + drain + flush ordering; app supplies domain hooks
try store.recover(.{
    .on_replayed = &Directory.recomputeCounts,   // was operations.recompute*
    .bootstrap   = &Directory.bootstrapRoots,     // was bootstrapRootCategories
});

// periodic workers: engine provides the thread + shutdown/cond/interval scaffolding;
// the app supplies the tick fn (was verifierLoop / repair_worker.loop)
var verifier_worker = store.spawnWorker(.{ .interval_ns = interval, .tick = &Directory.verifyOnce });
var repair_worker   = store.spawnWorker(.{ .interval_ns = tick,     .tick = &Directory.repairOnce });

// protocol: engine owns framing/pipelining/latency; app owns the op switch + statuses
zigstore.protocol.processFrames(ctx, conn, Directory.dispatch, &store.op_latency);

// server + bootstrap: generic over an opaque ctx + Handler + ServerConfig
try zigstore.run(Store, &directory, server_config);
```

The persisted monotonic counters (`next_category_id`/`next_link_id`/`next_repair_seq`) are
generated on `Store` as atomics from the schema's `.counters` list, persisted in
`Store.Header`, and reloaded on `init` — the app `Directory` reads and bumps them. The DMOZ
state that is in-memory and rebuilt on `recover` (`url_bloom`, `subtree_cache`, the
pending/approved/rejected status counts) lives in the app `Directory` struct that
**composes** `Store`.

## §4 · Blocker & coupling resolutions

The three blockers and the hardest couplings, with their fixes:

1. **`database.zig` → `StorageEngine`** *(blocker)*. The carve follows the struct's existing
   split: generic fields (file/header/cache/free_list/wal/snapshot/flusher/apply-ordering/
   `op_latency`) become `Store`; the DMOZ fields become either schema-generated (the named
   trees, the declared `.counters`) or app `Directory` state (bloom/subtree_cache/
   status-counts). Every
   engine→app import (`operations`, `verifier`, `subtree`, `repair`, `changeset`, `commit`,
   `wal_apply`) inverts into the `recover` hooks and `spawnWorker` ticks. **The engine
   imports none of them.**
2. **`binary_protocol.zig` framing split** *(blocker)*. `processFrames` (the pipelined frame
   loop: header parse, `op_latency[op_byte]` recording, response-buffer management) is fully
   generic once `dispatch` becomes an injected `dispatch_fn(ctx, op_byte, payload, count,
   resp) usize`. **Status:** the engine owns a base `Status { ok = 0, not_found = 1,
   duplicate = 2, invalid = 3, err = 4 }` (today's exact low codes); the app's
   `category_not_found = 5` / `has_children = 6` / `circular = 7` and all sub-statuses are
   app extensions, written to the wire as the raw `u8` the app handler returns.
3. **`file_header.zig` superblock** *(blocker)*. Resolved by the comptime-roots design in §3:
   `Store.Header` is generated to carry exactly the schema's roots/counts plus the generic
   fields, with an app-supplied `magic`.
4. **`types.zig` split** *(moderate, land first)*. The comptime toolkit moves to `codec.zig`
   (imports nothing from `src/` today — a clean leaf); the records and key wrappers stay in
   app `schema.zig`. This must land first because `changeset.zig`, `subtree.zig`, and
   `gen_client_ts.zig` all reach through `types.zig`.
5. **`changeset.zig` codec → `wire_codec.zig`** *(moderate)*. The `@typeInfo`-driven
   encode/decode is generic; the residual `Op`/`Effect`/`ChangeSet` union stays app-side and
   calls `wire_codec.encode(ChangeSet, …)`.
6. **`snapshot.zig` `anytype` → `SnapshotHost`** *(moderate)*. Replace `db: anytype` with a
   formal interface (`snapshot_in_progress`, `data_dir`, `wal_writer.getSequence`,
   `apply_mutex`, `mt_drain_mutex`, `cache.flushAll`, `flushHeader`, `header.page_count`),
   which the generated `Store` satisfies.
7. **`commit.zig` genericization** *(moderate)*. The engine owns the WAL-append +
   monotonic-seq ordering (seq N applies only after N−1, broadcast) + durability wait; the
   app supplies `serialize(cs)` and `apply(directory, cs)`.
8. **`gen_client_ts.zig` → `tsgen.zig`** *(moderate)*. The emitters lift; the field→TS-type
   and field→reader special-cases (`status` → `LinkStatus`, `created_at`/`updated_at` →
   `Date`) become a comptime table the app driver supplies.
9. **`wal` `OpCode` → plain `u8`** *(trivial)*. The engine namespaces nothing; the app owns
   record-kind codes.

## §5 · Conventions

`zigstore` gets its own `CLAUDE.md` + `.claude/rules`, adapted from `dmozdb`'s: `zig fmt`
clean, `defer`/`errdefer` pairing on every acquire, `std.testing.allocator` in tests,
return-errors-don't-panic, and "wire-format changes are protocol changes."

**Comment policy (overrides the global no-comment rule for this repo only):** `///` and
`//!` doc comments are allowed on **exported declarations** — the schema-plug API surface —
so Zig `autodoc`, IDE hovers, and third-party discoverability work. Internal/private code
keeps the strict discipline: no inline WHY comments, intent through naming and structure,
invariants encoded as named tests. `dmozdb`'s own rules are untouched.

## §6 · Build, packaging & dev workflow

- **Engine** (`zigstore/build.zig`): `b.addModule("zigstore", .{ .root_source_file =
  b.path("src/root.zig"), .target = target, .optimize = optimize })`. `root.zig` is a barrel
  that `pub const`-re-exports the surface (`Engine`, `schema`, `codec`, `wire_codec`,
  `protocol`, `EpollServer`, `run`, `ServerConfig`, `tsgen`) and aggregates the engine's own
  `test {}`. Test/exe steps take a `root_module` per Zig 0.15.
- **App** (`dmozdb/build.zig.zon`): a `zigstore` dependency. `build.zig` does
  `b.dependency("zigstore", …).module("zigstore")` and `exe_mod.addImport("zigstore", …)`.
  `gen_client_ts` and `dmoz_import` become app executables that `addImport("zigstore")`.
  `-Dcpu=baseline` threads through both `addModule` calls for Ampere/OKE.
- **Dev during the big-bang**: `zigstore` is a sibling checkout pinned via
  `.path = "../zigstore"` in `build.zig.zon` while migrating — atomic local edits, no tag
  churn. The flip to `url` + `hash` is the last step before the PR lands.

## §7 · Versioning & CI

- Separate `github.com/poly-glot/zigstore`, semver, first cut **`v1.0.0`**.
- **Engine CI**: `zig build test` on pinned **Zig 0.15.2 in a Linux container** (the WAL path
  needs `O_DIRECT`/`fallocate`, so the gate cannot run on the Mac host — the same constraint
  that already applies to `dmozdb`'s verify gate) plus a `-Dcpu=baseline` build.
- **Consume by tag**: `zig fetch --save=zigstore …/archive/refs/tags/vX.Y.Z.tar.gz` rewrites
  url + hash in one shot. Never hand-edit the hash. **Never move a published tag** — the hash
  is content-addressed, so a re-tag that changes the tree breaks every pin with a hash
  mismatch. MAJOR for any wire-format or public-API break; MINOR is additive only (extend ops
  with optional trailing bytes — never fork an op number). The app commit that bumps the
  engine commits `build.zig.zon` **and** the required call-site changes together, mirroring
  `dmozdb`'s "regenerate the client in the same commit" rule. Pre-release iteration can pin a
  commit-SHA tarball.

## §8 · Test strategy

- The engine repo owns the **infra tests that travel with the files** (page/btree/wal/cache/
  codec/framing/tokenizer roundtrips), moved out of `e2e_test.zig`, `types.zig`, and
  `changeset.zig` into the engine's `zig build test`.
- `dmozdb` keeps its DMOZ unit tests, Playwright e2e, and k6.
- **No single command tests the integrated stack** post-split — each repo tests its half. The
  integration gate is **`dmozdb`'s e2e + curl + a visual check of the rendered pages**, run on
  the migration branch against the `.path` engine *before* `v1.0.0` is tagged, and again after
  the dependency is flipped to the tagged tarball.

## §9 · Migration plan (one PR, ordered to stay compilable)

1. Split `types.zig` → `codec.zig` (engine) + `schema.zig` (app). **First** — everything
   reaches through it.
2. Lift `changeset.zig`'s codec → `wire_codec.zig`.
3. Move the 17 zero-back-edge files; repoint their imports.
4. Generate `Store.Header` from the schema; carve `StorageEngine`/`Store`; invert the
   engine→app imports into `recover` hooks + `spawnWorker` ticks.
5. Genericize `snapshot` (`SnapshotHost`) and `commit` (over a record type).
6. Lift `ServerConfig`, the generic `epoll` reactor (`Handler`), `protocol/framing` (dispatch
   callback + base `Status`), and the generic `run`.
7. Lift `tsgen` emitters; rebuild the app's codegen driver on top.
8. Repoint all app imports to `@import("zigstore")`; wire the `.path` dependency; build the
   app `Directory` composing `Store`, plus its `dispatch`/`Op`/statuses/hooks.
9. **Validate**: engine `zig build test`; app `zig build test`; app Playwright e2e
   (`--project=chromium`) + `curl` the routes + eyeball the rendered pages; k6 sanity.
10. Tag `zigstore` `v1.0.0`; flip the app dependency to tarball + hash; re-validate; land the
    single PR.

## §10 · Non-goals (v1.0)

- Pluggable VFS — stay Linux-only `O_DIRECT`/`fallocate`.
- Hierarchy/subtree/slug/breadcrumb logic — stays app-side.
- `searchCategories`/`searchLinks` — stay app-side; only the tokenizer/inverted-index
  primitive lifts.
- No io_uring/async rewrite.

## §11 · Risks & mitigations

- **Big-bang must land green at once.** Mitigated by the `.path` dev loop and the
  compilable-at-each-step migration order; the PR is reviewable as a sequence of commits.
- **Comptime-schema complexity.** The `Engine(schema)` generator is the bulk of the design
  work; it is bounded by `database.zig`'s existing `tree_fields`/`inline for` pattern, which
  it generalizes rather than invents.
- **Two-repo dance** for future cross-cutting changes — accepted cost of the separate-repo +
  semver requirement.
- **Hash-pinning friction** — mitigated by always using `zig fetch --save` and never moving a
  published tag.

## §12 · Decisions resolved during brainstorming

- Reuse scope: **BROAD** (full network-DB platform) — a concrete second consumer is intended.
- Sequencing: **big-bang to `v1.0.0`** as one validated PR.
- API style: **hybrid** (comptime data plane + runtime seams).
- Audience: **public / OSS-quality**; name **`zigstore`**.
- Comment policy: **doc comments on the public API only**.
- Superblock: **type-safe comptime roots** (not an opaque region).
- `wal` `OpCode`: **plain `u8`**.
- File backend: **Linux-only for v1.0** (pluggable VFS deferred).
